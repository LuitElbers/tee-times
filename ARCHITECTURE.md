# Architecture, Rate Limits & Scaling Notes

Hard-won lessons about how the tee-times backends behave, why the system is built
the way it is, and what will break when we add more courses. Read this before
touching the backends or adding ~100 more courses — most of these limits were
discovered the painful way and are easy to re-trip.

---

## 1. The core model: two tiers of backend

Every course is served by one of ~8 booking-system backends (`scrapers/*.py`).
They split into two groups, and **which group a backend is in is the single most
important fact about it**:

| Tier | Backends | How it's served | Why |
|---|---|---|---|
| **Fast** | intogolf, hollandschegolfclub, egolf4u, rijkvannunspeet, asparagi | Live, in the API request (`main.py`), capped at **8s** each | They answer fast enough to finish inside one request |
| **Slow / gated** | teecontrol, nexxchange, golfmanager (Waterland) | Pre-fetched out-of-band by the **warmer** → `warm.json` → `cache_backend.py` | They're rate-limited or token-gated and can't finish live |

If you add a course on a **fast** backend, it just works. If you add one on a
**slow** backend, it rides along with the warmer automatically (it's enumerated
in that scraper's `COURSES`). Adding a *new* slow backend means deciding how it's
served (live vs warmed) based on the limits below.

## 2. Per-backend rate-limit cheat sheet (measured, not guessed)

### teecontrol (`api.teecontrol.com`) — the strict one
- **Hard per-IP request-rate limit.** Bursts get `429`. A concurrent fan-out of
  all clubs at once (~123 requests) → **~70% got 429'd and were silently dropped**.
  This is what made teecontrol courses vanish originally.
- Measured safe rates (global throttle = min gap between request starts,
  `_MIN_INTERVAL` in `scrapers/teecontrol.py`):
  - `0.1s` (10/s) sustained over ~644 requests (warmer, 23 clubs × 7 days) → **429 storms**, ~32 failures, partial data.
  - `0.3s` (3.3/s) → **0 failures**, full coverage. This is the **warmer** rate.
  - `0.15s` (~7/s) → safe for a **single date** (~92 requests, ~28s). This is the **on-demand far-date** rate, set via `set_min_interval()` from `main.py`.
- **Retries amplify the problem.** When 429s happen, the retry backoffs make the
  run BOTH incomplete AND much slower (a 0.1s run took ~13min vs ~16min at 0.3s
  but complete — counterintuitively, going gentler was *barely* slower and far
  more reliable). **If you see 429s, slow down — don't add more retries.**
- **Probing can ban your own IP.** Hammering it during testing triggered an
  escalating cooldown that returned 429 even on `/auth/guest`. Test gently.
- Tokens cache 55min; a 60s response cache exists but is useless on serverless
  (each Vercel request is a cold process — the cache never warms across requests).

### nexxchange (`nexxchange.com`) — also rate-limited
- Rate-limits a concurrent fan-out too. Warmer uses `Semaphore(3)`; raising
  concurrency → 429s, fewer clubs succeed.
- ~28s for one date's 11 clubs. Separate host from teecontrol, so the two run
  concurrently without competing.

### golfmanager / Waterland — token-gated, needs a browser
- `availability.json` is gated by a **client-side signed `cid` header token**
  generated by their JS. Plain `httpx` gets **401**.
- The warmer fetches it with **Playwright (Chromium)** in GitHub Actions, waiting
  on the `availability.json` response (NOT `networkidle` — their SPA holds a live
  connection and never idles; `expect_response` is fast, networkidle wasted ~30s/date).
- **Cannot be fetched live on Vercel** (no browser). For far dates it's flagged
  with a visible notice, never silently dropped.

### Fast backends
- intogolf, hollandschegolfclub (the ITG/Proware cluster — many courses via one
  `location_id` endpoint), egolf4u, rijkvannunspeet, asparagi all answer within
  the 8s cap. No special handling needed.

## 3. Vercel platform facts (corrected — old assumptions were wrong)

- **Function time limit: ≥50s, not 10s.** The old code comments claimed "Vercel's
  default 10s limit" and capped backends at 8s because of it. **That was wrong.**
  Verified by deploying a temporary `/api/_sleeptest?secs=N` endpoint: requests
  up to **50s returned 200**. Hobby allows up to 60s. This is what makes the
  far-date live fetch (~40s) possible with no infra change.
  - To re-probe in future: add a temp sleep endpoint, hit it with increasing
    `secs`, watch for the first failure/504.
- **Deployment is configured in the Vercel dashboard**, not in the repo. There is
  **no `vercel.json` and no `api/` dir**. Don't assume you can add a `vercel.json`
  `functions` block to set `maxDuration` — if its glob matches no recognized
  function, the **build fails**. We never needed it (limit is already high enough).
- **`.gitignore` has `*.json`.** So `warm.json`, and any `vercel.json` you might
  add, are ignored — you must `git add -f` them. (`warm.json` is force-added to
  the `data` branch by the workflow.)
- Static files (`public/`) are served by Vercel's CDN; `main.py` only handles
  `/api/*` and the `/` → `/index.html` redirect.

## 4. The warmer (out-of-band pre-fetch)

- `.github/workflows/warm.yml` runs `warm.py` on a schedule (**every 20 min**) and
  on manual dispatch. It fetches the slow backends for **today..+6 (7 days)** at
  the widest settings (players=1, all holes, both course types) and force-pushes
  `warm.json` to an **orphan `data` branch** (single file, clean history).
- `cache_backend.py` reads `https://raw.githubusercontent.com/.../data/warm.json`
  (in-process 60s cache) and filters per request (free_slots, holes, is_short).
- **Run time ~16 min** at the gentle teecontrol rate. The 20-min interval is set
  *above* the run time on purpose (`cancel-in-progress: false`) so scheduled runs
  never pile up. If runs ever exceed the interval, they back up unboundedly.
- **The warmer runs in a separate process from Vercel.** That's why `main.py` can
  set a faster teecontrol rate (`0.15s`) for on-demand without affecting the
  warmer's safe `0.3s`.
- **Fetch per-club with bounded concurrency + retry, never a big concurrent
  gather.** The original all-clubs `asyncio.gather` was the burst that caused
  silent drops. `warm.py` records **per-club status** (`dates_ok`/`dates_failed`/
  errors) in `warm.json` — this is the seed for the (still-TODO) "broken courses"
  monitoring in the Banen view.
- GitHub Actions is **free & unlimited on public repos** (the repo was made public
  for this, which also lets the app read the raw `data`-branch URL without a token).
  If it ever goes private: 2000 free min/month, ~2 min/run → ~30-min interval max
  before it costs money.
- **`raw.githubusercontent.com` CDN-caches ~5 min.** To read the *latest* warm.json
  during testing, fetch the immutable commit-SHA URL
  `raw.githubusercontent.com/.../<sha>/warm.json` instead of the branch URL.

## 5. Far-date on-demand (beyond the warm window)

- The warmer only covers 7 days. For later dates, `main.py` (`_is_far_date`)
  live-fetches teecontrol + nexxchange for that single date (~40s, fits the 50s
  limit), instead of showing nothing. Waterland is flagged with a notice.
- Response shape is `{tee_times: [...], notices: [...]}`. Frontend shows a
  "duurt langer" loading message for far dates and renders notices in `#notice-row`.
- **Very far dates (>~2-3 weeks) legitimately return empty** for teecontrol/
  nexxchange — their booking windows aren't open yet. Not a bug; don't "fix" it.

## 6. Playbook: "a course stopped showing up"

1. **Run the scraper directly** (no API cap):
   `python -c "import asyncio; from scrapers.X import fetch_tee_times; print(len(asyncio.run(fetch_tee_times('YYYY-MM-DD',1,None,True,True))))"`.
   Works directly but missing in prod → it's a **timeout/rate-limit**, not a scraper bug.
2. **Check the warmer's per-club status** in `warm.json` (`status` field) — tells
   you which clubs failed and whether the errors are `429` (rate limit) or other.
3. **429s** → the rate is too high for the current volume; raise `_MIN_INTERVAL`
   (slower) or reduce load (fewer days / fewer clubs per run). Don't add retries.
4. **Timeouts** (backend > its cap) → the backend got slower or grew; it may need
   to move from live to warmed, or the warm run needs more time/headroom.
5. **Bypass the CDN cache** (commit-SHA URL) when checking if a fresh warm fixed it.

## 7. Scaling to +100 courses — what will break, in order

1. **The warmer run time is the first wall.** It's ~16 min for ~35 slow courses,
   dominated by teecontrol's gentle serial rate. Doubling the slow-backend course
   count roughly doubles the run → it will exceed the 20-min interval and back up.
   Fixes, cheapest first:
   - Reduce `DAYS` in `warm.py` (fewer look-ahead days).
   - Raise the cron interval above the new run time.
   - **Shard the warm** across parallel workflow runs (e.g. a matrix by backend or
     club-range) writing to separate files the reader merges — different clubs hit
     the same `api.teecontrol.com` IP-rate-limit though, so parallel teecontrol
     shards on the *same runner IP* won't help; they'd need different runners/IPs.
   - Move to a small **always-on worker + external store** (Redis/KV) that trickles
     refreshes continuously instead of big batches — the "proper" long-term answer.
2. **Far-date on-demand will exceed 50s** if many slow courses are near one user.
   Today it fetches *all* slow courses for the date (~40s). With far more slow
   courses, switch to **nearby-only** (the browser knows user + course coords) or
   chunk the fetch across multiple requests with progressive rendering.
3. **Fast backends scale fine** as long as each finishes < 8s and the whole live
   request stays < 50s. Adding many fast-backend courses is low-risk.
4. **New rate-limited backends**: measure the safe rate *before* wiring them in
   (probe gently — see §2), and decide live-vs-warmed by whether one date's worth
   of requests fits comfortably under ~30s.

## 8. Traps that cost time here (don't repeat them)

- `*.json` is gitignored → `git add -f` for `warm.json`/`vercel.json`.
- A `vercel.json` `functions` glob matching nothing **fails the build**.
- Probing a rate limit aggressively **bans your own IP** temporarily.
- A big concurrent `gather` over rate-limited clubs **silently drops** the ones
  that 429 — always bounded concurrency + per-item status.
- Playwright `networkidle` hangs on SPAs with live connections — use `expect_response`.
- Desktop Chromium ≠ iOS Safari for native form controls (see the iOS memory).
- `raw.githubusercontent` branch URLs serve stale (~5 min) — use commit-SHA URLs to verify fresh data.
